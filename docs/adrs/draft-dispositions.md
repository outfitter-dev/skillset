# Draft ADR Dispositions

This table is the factual input to
[SET-325](https://linear.app/outfitter/issue/SET-325/promote-only-implementation-proven-draft-adr-chains).
It records the binding SET-327 audit of every draft present on 2026-07-21. It
does not promote a draft by itself. Promotion still requires the stated
amendment, dependency order, current verification, and ADR lifecycle tooling.

The audit distinguishes a narrow amendment, which preserves the decision's
identity while correcting a bounded part, from whole-decision supersession,
which preserves the old record as history and accepts a successor.

## SET-325 lifecycle application

For this audit, **Promote as written** preserves the decision and body; it does
not preserve stale lifecycle metadata. Before promotion, SET-325 must normalize
`depends_on` to its binding backward-only numeric graph. Core Library and CLI
Boundary and Source Manifest Listing Metadata must each use `depends_on: [0,
1]`. Forward or cyclic dependency edges are invalid.

## Binding SET-325 order

1. Independent foundation after dependency normalization: Core Library
   Boundary; Feature Reference and Schema Registry; Agent Source Model; Source
   Manifest Listing Metadata.
2. Workspace lineage: amend/promote Skillset Workspace Layout; mark Unified
   Source Layout superseded; promote Named Partials; amend/promote Source Test
   Selection; amend/promote Fixtures, Tests, Dogfooding, and Evals.
3. Change lineage: amend/promote Source Change and Release Provenance; mark
   Changelog and Versioning superseded; amend/promote Reason-Only Change Ledger;
   amend/promote Change and Release Edge Decisions.
4. Render lineage: author/promote the full Render Results successor; mark
   Lowering Outcomes superseded; preserve ADR-0003 as the narrow policy
   amendment; amend/promote Deterministic Projection and Adapter Conformance.
5. Tools lineage: narrow/retitle/promote Portable Agent Authority Intent as the
   portable skill tools policy; amend/promote Post-Tools Policy Boundary.
6. CLI lineage: amend/promote Workflow-Oriented CLI; Versioned Structured Output;
   One-Action Repo Adoption.
7. Leave First-Class Sets, Global XDG, Model Aliases, Reviewed Settings, YAML
   Formatting, Path References/Rename, and Catalog Refresh unpromoted.

## Rejected alternatives

- Do not reduce the global/XDG draft to non-activation doctrine and call the
  central root/authority decision accepted.
- Do not hold proved change-edge decisions behind an unimplemented baseline
  command or create two whole successors for early changelog/versioning.
- Do not treat removal of standalone `adopt` as whole supersession when the
  repo-local one-action invariant survives as `init --adopt`.
- Do not wait for complete Core extraction or claim that a reviewed spec-only
  decision is already implemented.
- Do not rewrite the historical lowering-outcomes or unified-layout drafts;
  preserve them and accept whole successors.
- Do not create public per-command schemas merely to preserve stale draft prose,
  or claim Skillset controls enclosing Codex/Cursor authority.
- Do not treat ADR-0003 as the whole post-tools or Render Results successor.
- Do not duplicate SET-269 or create speculative current-drift issues for sets,
  model aliases, general settings workflow, behavioral evals, or broad provider
  enforcement.
